The biggest surprise: your architecture is actually well-designed.
Look at what you already have:

"Conversational" intent category exists in the prompt with the exact failing example
Fallback patterns are well-designed and would catch greetings
after_query_analysis already skips retrieval for conversational intents
after_confidence already bypasses silence/hedge checks for conversational intents
The LangGraph conditional edges are wired correctly

The entire skip-retrieval pathway is built and works. It just never fires because of one upstream failure: the LLM returns valid JSON with the wrong intent class, and your fallback only triggers on parse errors.
This means the fix is much more surgical than I initially thought. You don't need to rearchitect the pipeline into agentic tool-use. The pipeline design is sound — it's the gateway into the pipeline that's broken.

But — and this is important — fixing the intent classifier alone gets you maybe 40% of the way. Here's why:
Even once "hi I'm from India" correctly routes to conversational mode and skips retrieval, the generation prompt is still the same thin 2-sentence bio with "You MUST cite your sources." So when a user asks a real question and the pipeline works correctly, the response will still feel like a search engine, not like Parag Khanna. The Delphi comparison exposed both problems — broken routing AND shallow persona.
So I'd frame this as two fixes, not one:
Fix A (the gate): Make intent classification reliable. This unblocks the already-built conversational pathway.
Fix B (the soul): Give the clone a real persona so even RAG-backed responses feel human.

Let me go deep on Fix A, because there are multiple approaches and they have different tradeoffs:
Approach A1: Dual-validation (patch the existing flow)
After the LLM returns its JSON classification, run the regex fallback patterns always — not just on parse error. If the regex says "conversational" but the LLM says "factual," trust the regex for high-confidence pattern matches (greetings, personal statements) and trust the LLM for ambiguous cases. This is a ~20 line code change — move the fallback block outside the except clause and add an override condition.
The advantage: minimal change, ships in an hour. The risk: you're papering over the real problem (LLM misclassification) with heuristics. Edge cases will leak through — "hi, can you explain what you think about India's infrastructure?" is a greeting AND a substantive question. Regex won't handle that well.
Approach A2: Two-pass classification
First pass: cheap regex/keyword check. If it's clearly conversational (greeting words, no question marks, no knowledge-seeking words), set intent and skip the LLM call entirely — saves latency too. Second pass: for everything else, use the LLM classification as today.
This is more robust than A1 because it doesn't rely on the LLM for cases that don't need it. But it still has the same edge case problem with hybrid messages.
Approach A3: Fix the LLM classification itself
The prompt has the right example, but Qwen3.5 still misclassifies. This could be because the prompt is too long and the example gets buried, or because the model is biased toward information-seeking classifications. You could restructure the prompt to make conversational detection the first check — "Before classifying intent, determine: is this message primarily social/conversational? If yes, classify as conversational and stop." Making it a binary gate before the 6-way classification.
You could also add a confidence field to the JSON output — ask the LLM how confident it is in the classification. Low confidence + greeting-like patterns → override to conversational.
My recommendation: combine A2 and A3. Regex pre-filter for obvious cases (saves an LLM call), then improved prompt for ambiguous cases. The fallback patterns you already have are good — they just need to run first, not last.

Now for Fix B — the persona problem. This is where I want to push your thinking.
Your current generation prompt:

"You are Parag Khanna. About you: Author, geopolitical strategist known for synthesizing complex global trends into accessible frameworks. Data-driven analysis connecting history, economics, and geography."

Compare this to what a rich persona prompt should contain. Think of it in layers:
Identity layer — who is he? Born in Kanpur, India. Grew up across India, UAE, Germany, United States. Lives in Singapore. Father of two. Founded FutureMap and Climate Alpha. Advisor to governments. These are facts the clone should know about itself without retrieving them.
Voice layer — how does he talk? Direct, optimistic but evidence-based, uses frameworks and taxonomies, frequently draws analogies across regions, asks questions back, comfortable disagreeing, uses phrases like "the data shows" and "what the map tells us." Read 10 of his interviews and you'd extract 20+ style patterns.
Relational layer — how does he interact? Curious about where the other person is from, connects their context to global trends, doesn't lecture — converses, acknowledges what he doesn't know with confidence rather than hedging nervously, makes the other person feel like their local perspective matters globally.
Boundary layer — what does he refuse to do? Won't make specific political predictions, won't endorse candidates, distinguishes analysis from advocacy, redirects personal gossip back to structural analysis.
None of this is in your prompt. And critically, the generation prompt needs to be intent-aware. For conversational messages, the instruction should be "respond naturally as Parag, draw on your identity and relational style, no citations needed." For substantive questions, "draw on the provided context, cite sources, but weave them in naturally — you're having a conversation, not writing a footnoted essay."

The difference between a good persona prompt and a great one is whether it's based on how the person writes (books/essays) or how they talk (interviews, podcasts, Q&A sessions).


One more thing that jumped out:
Your Mem0 injection — "Relevant context from memory:\n{memory}" appended to the user message — is doing almost nothing useful. The LLM has no idea what to do with this. It needs to be framed as: "You remember the following about this person from previous conversations: [memory]. Use this to personalize your response — reference prior discussions naturally, like a friend would."
And it should be injected in the system prompt, not the user message. Putting it in the user message means the LLM might treat it as something the user said, not something the clone remembers.
