Layer 3: The Scoring Inversion (why the system thinks it's doing well)
This one is subtle but critical. Your confidence scorer gives 79% to that broken response. That means your monitoring dashboard, your analytics, your evaluation framework — they all say "system is performing well" when it's actually producing garbage. You're flying blind on quality.
The 4 factors (retrieval confidence, citation coverage, response grounding, passage count) all measure the same thing: "did the response use retrieved content?" For a conversational message, using retrieved content is the wrong thing to do. A perfect conversational response would score near zero on all 4 factors.
Question for codebase:

Does the confidence score get stored in query_analytics? If so, you might be able to pull historical data and see how many queries were actually conversational but got scored as if they were substantive. That would tell you the scale of the problem.


What I think the research direction should be before writing SOLVE.md:
Before jumping to solutions, I think you need to study a few things:
A. How do the best persona products handle mode-switching? Delphi, Character.ai, Inflection Pi — they all seamlessly shift between casual chat and domain knowledge. The common pattern seems to be that retrieval is a tool the persona can choose to use, not a mandatory pipeline step. This is fundamentally different from your architecture where every query goes through the same 5 steps.
B. What does a rich persona prompt actually look like? Not just bio — but conversational rules, example exchanges, identity facts, style guidelines. Some of the Character.ai community has shared persona prompt structures. The key insight is that you need enough identity in the prompt that the LLM can handle 70% of conversations without retrieval.
C. Agentic RAG vs Pipeline RAG. Your current architecture is pipeline RAG — every query follows the same path. The more modern approach is agentic RAG where the LLM itself decides whether to retrieve, and if so, what to retrieve. LangGraph supports this pattern natively with tool-calling nodes. This would be a bigger architectural change but would fundamentally solve the "retrieval fires when it shouldn't" problem.
